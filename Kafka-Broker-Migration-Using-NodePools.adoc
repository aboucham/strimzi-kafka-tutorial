= Tutorial: Zero-Downtime Broker Migration using Node Pools and Cruise Control
:toc: left
:icons: font
:source-highlighter: highlightjs

== Introduction
This guide demonstrates how to migrate Kafka brokers between node pools to transition from unsupported storage types (NFS) to supported block storage (Ceph) without cluster downtime. We utilize `KafkaNodePool` resources for lifecycle management and `Cruise Control` for safe data migration.

== Prerequisites
* Red Hat Streams for Apache Kafka 3.0 (Strimzi 0.46.x) Cluster Operator installed.
* An OpenShift 4.14 to 4.19 cluster.
* `oc` CLI tool configured with cluster-admin or strimzi-admin permissions.

== Phase 1: Deploy Initial Kafka Cluster
Deploy a KRaft-enabled cluster into the `strimzi` namespace. This cluster starts with brokers on IDs 0, 1, and 2.

[source,bash]
----
# Create and switch to the namespace
oc new-project strimzi
oc project strimzi

# Deploy the KRaft Kafka cluster
oc apply -f https://raw.githubusercontent.com/aboucham/strimzi-kafka-tutorial/refs/heads/main/kafka/kafka-cluster-kraft-full.yaml
----

=== Check Deployment Status
Verify that the initial brokers (IDs 0, 1, 2) and controllers (IDs 3, 4, 5) are running.

[source,bash]
----
oc get pods -n strimzi
----

== Phase 2: Enable Cruise Control
Enable Cruise Control to manage the data movement. This triggers a rolling update of all Kafka brokers to enable the Metrics Reporter.

[source,bash]
----
oc patch kafka kafka-cluster -n strimzi --type=merge -p '{"spec": {"cruiseControl": {}}}'
----

== Phase 3: Create Test Data
Create three topics to verify the migration. Initially, these partitions will be hosted on the original brokers.

[source,bash]
----
# Create test topics
for i in {0..2}; do
  oc run provision-test-$i -ti --image=registry.redhat.io/amq-streams/kafka-40-rhel9:3.0.1 --rm --restart=Never -- \
    bin/kafka-topics.sh --bootstrap-server kafka-cluster-kafka-bootstrap:9092 --create --topic topic-$i --partitions 1 --replication-factor 1
done

# Verify current partition placement
oc run -ti --verify-initial --image=registry.redhat.io/amq-streams/kafka-40-rhel9:3.0.1 --restart=Never -- \
  bin/kafka-topics.sh --bootstrap-server kafka-cluster-kafka-bootstrap:9092 --describe
----

**Expected Initial State:**
* `topic-0` -> Leader: **0**
* `topic-1` -> Leader: **1**
* `topic-2` -> Leader: **2**

== Phase 4: Create Supported Node Pool (`pool-b`)
Define the new pool using Ceph block storage. Use the `strimzi.io/next-node-ids` annotation to assign IDs 6, 7, and 8 to the new brokers.

[source,yaml]
----
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaNodePool
metadata:
  name: pool-b
  labels:
    strimzi.io/cluster: kafka-cluster
  annotations:
    strimzi.io/next-node-ids: "[6-8]"
spec:
  replicas: 3
  roles:
    - broker
  storage:
    type: jbod
    volumes:
      - id: 0
        type: persistent-claim
        size: 100Gi
        class: ceph-block-storage # Use your Ceph storage class name
----

Apply the new pool:
[source,bash]
----
oc apply -f pool-b.yaml
----

== Phase 5: Reassign Partitions (The Migration)
Move data from the old brokers (0, 1, 2) to the new brokers (6, 7, 8) using a `KafkaRebalance` resource in `remove-brokers` mode.

[source,yaml]
----
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaRebalance
metadata:
  name: migrate-data
  labels:
    strimzi.io/cluster: kafka-cluster
spec:
  mode: remove-brokers
  brokers: [0, 1, 2]
----

=== Execute and Approve
[source,bash]
----
# Apply the rebalance
oc apply -f rebalance.yaml

# Wait for status to reach 'ProposalReady'
oc get kafkarebalance migrate-data -o wide -w

# Approve the optimization proposal
oc annotate kafkarebalance migrate-data strimzi.io/rebalance=approve
----

== Phase 6: Verification and Cleanup
Verify that all data has successfully moved to the new pool.

[source,bash]
----
# Final check of partition placement
oc run -ti --verify-final --image=registry.redhat.io/amq-streams/kafka-40-rhel9:3.0.1 --restart=Never -- \
  bin/kafka-topics.sh --bootstrap-server kafka-cluster-kafka-bootstrap:9092 --describe
----

**Expected Final State:**
* `topic-0` -> Leader: **6**
* `topic-1` -> Leader: **7**
* `topic-2` -> Leader: **8**

=== Delete Old Pool
Once the data is safe, remove the NFS-backed node pool.

[source,bash]
----
oc delete kafkanodepool pool-a
----

The Cluster Operator will automatically delete the old broker pods and clean up the associated resources.
